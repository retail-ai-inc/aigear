# AIGear Project Full Guide

> [!NOTE]
> This document is a **complete development and deployment guide** for machine learning projects generated based on the AIGear standard template engine `aigear-init`.
>
> 💡 For a quick reference to the configuration parameter dictionary, please refer to the [Configuration Guide](route-guide.md). This document focuses on explaining **project architecture, directory operational mechanisms, deployment principles, and the use of core APIs**.

---

## 1. Core Architecture Design

AIGear is designed to provide a standardized ML / Recommendation System microservice framework that can be developed locally and seamlessly integrated with GCP (Google Cloud Platform) or Kubernetes (K8s) environments.

Key design principles include:
- **Separation of Configuration and Code**: Core environment variables are stripped into `env.json` or Secret Manager.
- **Environment Isolation**: Supports seamless switching between `local`, `staging`, and `production` environments.
- **Decoupling of Phases**: Standard machine learning business logic is abstracted into extensible pipeline stages: `fetch_data`, `preprocessing`, `training`, and `model_service`.
- **Minimalist Deployment**: Supports packaging and publishing trained models directly as independent gRPC microservices via `aigear-deploy-model` or Cloud Build.
- **Dual-Container Design**: Separates the training pipeline container (`Dockerfile.pl`) from the model serving container (`Dockerfile.ms`) for independent versioning and deployment.

## 2. Directory Structure Details

A standard project generated by `aigear-init --name my_service --pipeline_versions v1` has the following directory structure:

```text
my_service/
├── cloudbuild/               # GCP Cloud Build CI/CD configuration folder
│   └── cloudbuild.yaml       # CI/CD build and release control script
├── docs/                     # Project documentation library
├── kms/                      # Mount point for key management or certificates
├── src/                      # Core source code directory
│   └── pipelines/            # Core business logic implementation layer
│       └── v1/               # Pipeline version isolation layer
│           ├── fetch_data/   # Data fetching logic
│           ├── preprocessing/# Data preprocessing and feature engineering
│           ├── training/     # Main loop for model training
│           └── model_service/# Model serving logic and gRPC node
├── Dockerfile.pl             # Pipeline container (fetch_data → preprocessing → training)
├── Dockerfile.pl.dockerignore
├── docker-compose-pl.yml     # Orchestration for the pipeline container
├── requirements_pl.txt       # Pipeline Python dependencies
├── Dockerfile.ms             # Model service container (gRPC inference server)
├── Dockerfile.ms.dockerignore
├── docker-compose-ms.yml     # Orchestration for the model service container
├── requirements_ms.txt       # Model service Python dependencies
├── env.sample.json           # Environment configuration reference (copy to env.json)
└── README.md
```

> **Multiple pipeline versions:** Use `--pipeline_versions v1,v2` to scaffold multiple version directories under `src/pipelines/` simultaneously.

### 2.1 Structural Workflow Analysis

- **Pipeline Hierarchical Division**: Under `src/pipelines/`, versions are divided by directory name (e.g., `v1`, `v2`), supporting multiple algorithm experimental workflows within the same repository.
- **Single Responsibility Flow (Data → Feature → Train → Serve)**:
  1. `fetch_data`: Docks with MongoDB / BigQuery, abstracting the I/O layer.
  2. `preprocessing`: Feature flattening, aggregation, and anomaly removal, outputting standard Numpy/Tensors.
  3. `training`: Pure model core (early stopping, hyperparameter search, etc.).
  4. `model_service`: Model serving endpoint. When deployed via `aigear-deploy-model` or `aigear-grpc`, this module handles gRPC inference requests.
- **Two Dockerfiles**: `Dockerfile.pl` runs the training pipeline steps; `Dockerfile.ms` runs the gRPC model service. They have separate dependency files (`requirements_pl.txt` vs. `requirements_ms.txt`) and can be versioned independently.

## 3. Advanced Deployment Principles

Although the tables in `route-guide.md` enumerate all configurations, their actual linkage effects in engineering are as follows:

### 3.1 GCP-Based Build Pipeline (Cloud Build)
When `env.json` has `aigear.gcp.cloud_build.on = true`, and the branch matches the interception push via `branch_pattern` (`^main$`):
- The GCP trigger will read the source file pool.
- Execute `cloudbuild.yaml`. Inject actual environment context using `substitutions` (e.g., `_ENV`, `_REGION`).
- Run tests and ultimately package `Dockerfile.pl` or `Dockerfile.ms` into the configured Artifacts repository.

### 3.2 gRPC Inference Service Deployment

AIGear supports two deployment targets for the gRPC model service:

**Local (Docker Compose):**
```bash
aigear-deploy-model --version v1 --model_class_path src.pipelines.v1.model_service.MyModel
```

**GCP (Kubernetes):**
```bash
aigear-deploy-model --version v1 --model_class_path src.pipelines.v1.model_service.MyModel --gcp
```

To start the gRPC server directly in the current process (e.g., inside a container entrypoint):
```bash
aigear-grpc --version v1 --model_class_path src.pipelines.v1.model_service.MyModel
```

To delete a deployment:
```bash
aigear-deploy-model --model_class_path src.pipelines.v1.model_service.MyModel --delete
aigear-deploy-model --model_class_path src.pipelines.v1.model_service.MyModel --gcp --delete
```

When `multi_processing.on = true` is configured in `env.json`, the gRPC server will run in multi-processing mode with the specified `process_count` of workers. The Kubernetes deployment is subject to HPA auto-scaling influenced by the `min_nodes/max_nodes` parameters.

### 3.3 Monitoring and Tracking (Sentry & Logging)
- With `sentry.on = true`, the project will read the `dsn` configuration. All global Exceptions and gRPC request errors that are not caught by the `src` layer will be automatically reported to Sentry, where `traces_sample_rate` controls the granularity of tracking details.
- Setting `logging = true` forcibly starts the detailed DEBUG-level GCP Cloud Logging stream controlled by Aigear.

## 4. Aigear Utility Libraries (API Reference)

To accelerate the writing of business code, the AIGear framework has built-in strictly encapsulated cloud-native tool layers (relying on `aigear.*` modules) for developers. Development conventions require prioritizing the use of built-in Clients to access foundational platforms to avoid reinventing the wheel.

### 4.1 Configuration Management

**Function**: Hierarchically reads `env.json` and performs objectified parsing.

```python
from aigear.common.config import EnvConfig

config = EnvConfig.read_config()

# Objectified invocation, avoiding hard-coded nested dictionary retrieval
project = config.aigear.gcp.gcp_project_id
is_grpc_enabled = config.pipelines.v1.model_service.release_grpc
```

Auto-generate a Pydantic schema from your `env.json`:
```bash
aigear-env-schema
# Force regenerate
aigear-env-schema --force
```

### 4.2 Cloud Storage Bucket

**Function**: Minimalist workflow encapsulation for GCP Cloud Storage Buckets.

```python
from aigear.bucket import BucketClient

bucket_client = BucketClient(
    project_id="my-gcp-project",
    bucket_name="my-ml-bucket",
    bucket_on=True
)

bucket_client.download(
    bucket_blob_name="models/model_v1.pth",
    local_blob_name="model.pth",
    local_blob_path="/tmp"
)

bucket_client.upload(
    local_blob_name="/tmp/metrics.csv",
    bucket_blob_name="outputs/metrics.csv"
)
```

### 4.3 Database Access Pooling (MongoDB)

**Function**: Manages the MongoDB connection pool and securely integrates GCP password locks.

```python
from aigear.mongodb import MDBClient

mdb_client = MDBClient(project_id="my-gcp-project")

# [Security Preferred] Use GCP Secret Manager to extract URI strings
db = mdb_client.connect_db_by_secret(
    mongo_uri_secret="TRIAL_MONGO_URI",
    db_name_secret="MONGO_DB_NAME"
)

collection = db["users"]
results = collection.find({"is_active": True})
```

### 4.4 Extended Cloud Logging

**Function**: Smoothly integrates local standard output debugging with the online structured (JSON Payload) log collection system.

```python
from aigear.cloud_logging import Logging, wrap_logger

logger_controller = Logging(
    log_name="my-pipeline-log",
    project_id="my-gcp-project"
)

# Only print to console in development environment
logger = logger_controller.console_logging()
# Archive to Cloud Logging in online environment
# logger = logger_controller.cloud_logging()

# Add module prefixes to different layers via wrap
train_logger = wrap_logger(logger, prefix="[TRAIN] ")
train_logger.info("Epoch 1/35 completed successfully.")
```

### 4.5 GCP Secret Management

**Function**: Strips high-risk fields such as passwords/tokens.

```python
from aigear.secretmanager import SecretManager

secrets = SecretManager(project_id="my-gcp-project")

client_secret = secrets.get_secret_val(
    secret_id="MY_CLIENT_SECRET",
    secret_version="latest"
)
```

### 4.6 Third-Party Push Bridge (Slack)

**Function**: Sends operational notifications based on execution status.

```python
from aigear.slack import SlackNotifier

notifier = SlackNotifier(
    webhook_url="https://hooks.slack.com/services/YOUR/WEBHOOK",
    slack_on=True
)

notifier.slack_webhook(
    message_content="Pipeline v1 fetch_data finished with 10k rows.",
    system_name="DataFetcher",
    pipeline_name="v1",
    step_name="fetch_data",
    event_type="success"  # success / warning / failure
)
```

### 4.7 Running Pipeline Steps

**Function**: Execute a single named pipeline step by loading it as a Python module. The step is specified as a dotted module path, and the function receives `pipeline_version` as an argument.

**Via CLI (recommended):**
```bash
aigear-workflow --version v1 --step src.pipelines.v1.fetch_data.run
aigear-workflow --version v1 --step src.pipelines.v1.preprocessing.run
aigear-workflow --version v1 --step src.pipelines.v1.training.run
```

**Step module convention** — each step module should expose a callable that accepts `pipeline_version`:
```python
# src/pipelines/v1/training/run.py
def run(pipeline_version: str):
    # training logic here
    ...
```

**Via Python (e.g., inside a container entrypoint):**
```python
from aigear.common.loading_module import LoadModule

module = LoadModule("src.pipelines.v1.training.run").load_module()
module(pipeline_version="v1")
```

## 5. Complete Deployment Workflow

A typical end-to-end workflow from project initialization to production deployment:

```bash
# 1. Scaffold the project
aigear-init --name my_ml_service --pipeline_versions v1

cd my_ml_service

# 2. Configure env.json
cp env.sample.json env.json
# ... edit env.json with your GCP project, bucket, etc. ...

# 3. Generate Pydantic schema for config validation (optional)
aigear-env-schema

# 4. Create all GCP infrastructure
aigear-gcp-infra

# 5. Build Docker images
aigear-image                  # builds both Dockerfile.pl and Dockerfile.ms
aigear-image --push           # build and push to Artifact Registry

# 6. Run a one-off pipeline step locally
aigear-workflow --version v1 --step src.pipelines.v1.training.run

# 7. Schedule recurring pipeline runs on GCP
aigear-scheduler --version v1 --step_names fetch_data,preprocessing,training

# 8. Deploy the model serving gRPC service
# Locally:
aigear-deploy-model --version v1 --model_class_path src.pipelines.v1.model_service.MyModel
# On GCP:
aigear-deploy-model --version v1 --model_class_path src.pipelines.v1.model_service.MyModel --gcp
```
