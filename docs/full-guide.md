# AIGear Project Full Guide

> [!NOTE]
> This document is a **complete development and deployment guide** for machine learning projects (such as `my_service`) generated based on the AIGear standard template engine `aigear-init`.
>
> 💡 For a quick reference to the configuration parameter dictionary, please refer to the [Configuration Guide](route-guide.md). This document focuses on explaining **project architecture, directory operational mechanisms, deployment principles, and the use of core APIs**.

---

## 1. Core Architecture Design

AIGear is designed to provide a standardized ML / Recommendation System microservice framework that can be developed locally and seamlessly integrated with GCP (Google Cloud Platform) or Kubernetes (K8s) environments.

Key design principles include:
- **Separation of Configuration and Code**: Core environment variables are stripped into `env.json` or Secret Manager.
- **Environment Isolation**: Supports seamless switching between `local`, `staging`, and `production` environments.
- **Decoupling of Phases**: Standard machine learning business logic is abstracted into extensible Node pipeline stages such as `fetch_store_list`, `fetch_data`, `preprocessing`, `training`, and `release`.
- **Minimalist Deployment**: Supports packaging and publishing trained models directly as independent gRPC microservices via Cloud Build.

## 2. Directory Structure Details

A standard project generated by `aigear-init` typically has the following directory structure:

```text
my_service/
├── cloudbuild/           # GCP Cloud Build CI/CD configuration folder
│   └── cloudbuild.yaml   # CI/CD build and release control script
├── docs/                 # Project documentation library (includes route-guide.md, full-guide.md)
├── kms/                  # (If any) Mount point for sensitive materials like key management or certificates
├── src/                  # Core source code directory
│   ├── common/           # Common functional modules reused across Pipelines (config reading, logging, common tools)
│   └── pipelines/        # Core business logic implementation layer
│       └── pipeline_version_1/ # Pipeline instance isolation layer
│           ├── fetch_data/     # Data fetching logic
│           ├── preprocessing/  # Data preprocessing and feature engineering
│           ├── training/       # Main loop for model training
│           └── release/        # Model packaging and gRPC node related processing
├── Dockerfile            # Build specifications for containerizing the project runtime environment
├── docker-compose.yml    # Orchestration script for quickly pulling up local dependencies (e.g., databases/caches)
├── env.sample.json       # Environment configuration reference file (must be renamed to env.json for local execution)
└── requirements.txt      # Python core dependency lock
```

### 2.1 Structural Workflow Analysis

- **Pipeline Hierarchical Division**: Under `src/pipelines/`, versions are divided by `pipeline_version_x`, supporting the execution of multiple algorithm experimental workflows within the same code repository.
- **Single Responsibility Flow (Data -> Feature -> Train -> Release)**:
  1. `fetch_data`: Responsible for docking with MongoDB / BigQuery, abstracting the I/O layer.
  2. `preprocessing`: Responsible for feature flattening, aggregation, and anomaly removal, outputting standard Numpy/Tensors.
  3. `training`: The pure model core (mounting point for logics like early stopping, hyperparameter search, etc.).
  4. `release`: Determines the destination of the output. When `release_grpc: true` is configured in `env.json`, this step may trigger/orchestrate the deployment of an online inference service for the model artifacts.

## 3. Advanced Deployment Principles

Although the tables in `route-guide.md` enumerate all configurations, their actual linkage effects in engineering are as follows:

### 3.1 GCP-Based Build Pipeline (Cloud Build) 
When `env.json` has `aigear.gcp.cloud_build.on = true`, and the branch matches the interception push via `branch_pattern` (`^main$`):
- The GCP trigger will read the source file pool.
- Execute `cloudbuild.yaml`. Inject actual environment context using `substitutions` (e.g., `_ENV`, `_REGION`).
- Run tests and ultimately package the `Dockerfile` into the configured Artifacts repository (`test-pipelines-image-whn`).

### 3.2 Kubernetes gRPC Inference Service
If the business scenario requires real-time online evaluation, and `pipelines.xxx.release.grpc` is configured:
- The aigear runner or pipeline will enable a gRPC endpoint (listening on `port: 50051`, `service_host: 0.0.0.0`).
- When `multi_processing.on = true` is enabled, the gRPC server will run in a multi-processing mode with the specified `process_count` of Workers to squeeze node performance.
- When the container is spun up or down in the Kubernetes cluster (`my-grpc-cluster`), it will load the pre-configured `model_path` (usually corresponding to a VPC/NAS mount point, or the `LightSANs.pth` artifact just downloaded via Bucket).
- This Kubernetes mechanism is also subject to HPA auto-scaling influenced by the `min_nodes/max_nodes` parameters.

### 3.3 Monitoring and Tracking (Sentry & Logging)
- With `sentry.on = true`, the project will read the `dsn` configuration. All global Exceptions and gRPC request errors that are not caught by the `src` layer will be automatically reported to Sentry, where `traces_sample_rate` controls the granularity of tracking details.
- Setting `logging = true` forcibly starts the detailed DEBUG-level GCP Cloud Logging stream controlled by Aigear.

## 4. Aigear Utility Libraries (API Reference)

To accelerate the writing of business code, the AIGear framework has built-in strictly encapsulated cloud-native tool layers (relying on `aigear.*` modules) for developers. Development conventions require prioritizing the use of built-in Clients to access foundational platforms to avoid reinventing the wheel.

### 4.1 Configuration Management (config.py)

**Function**: Hierarchically reads `env.json` and performs objectified parsing.

```python
from aigear.config import read_config

config = read_config()

# Objectified invocation, avoiding hard-coded nested dictionary retrieval
project = config.gcp.gcp_project_id
is_grpc_enabled = config.pipelines.pipeline_version_1.release.release_grpc
```

### 4.2 Cloud Storage Bucket (bucket.py)

**Function**: Minimalist workflow encapsulation for GCP Cloud Storage Buckets.

```python
from aigear.bucket import BucketClient

# Instantiate bucket communication protocol
bucket_client = BucketClient(
    project_id="ssc-ape-staging",
    bucket_name="medovik-xxx-staging",
    bucket_on=True
)

# Direct operations
bucket_client.download(
    bucket_blob_name="models/LightSANs_v1.pth",
    local_blob_name="LightSANs.pth",
    local_blob_path="/tmp"
)

bucket_client.upload(
    local_blob_name="/tmp/metrics.csv",
    bucket_blob_name="outputs/metrics.csv"
)
```

### 4.3 Database Access Pooling (mongodb.py)

**Function**: Manages the MongoDB connection pool and securely integrates GCP password locks.

```python
from aigear.mongodb import MDBClient

mdb_client = MDBClient(project_id="ssc-ape-staging")

# [Security Preferred] Use GCP Secret Manager to extract URI strings for direct connection
db = mdb_client.connect_db_by_secret(
    mongo_uri_secret="TRIAL_MONGO_URI",
    db_name_secret="MONGO_DB_NAME"
)

# Execute standard pymongo syntax
collection = db["users"]
results = collection.find({"is_active": True})
```

### 4.4 Extended Cloud Logging (cloud_logging.py)

**Function**: Smoothly integrates local standard output debugging with the online structured (JSON Payload) log collection system.

```python
from aigear.cloud_logging import Logging, wrap_logger, set_log_preprocessor

# Initialize the log object in dual or single mode
logger_controller = Logging(
    log_name="test_sklearn_pipeline_log",
    project_id="ssc-ape-staging"
)

# Only print to console in development environment
logger = logger_controller.console_logging() 
# Archive to Cloud Logging console in online environment
# logger = logger_controller.cloud_logging()

# Add module prefixes to different layers via wrap, making it easier to track
train_logger = wrap_logger(logger, prefix="[TRAIN] ")
train_logger.info("Epoch 1/35 completed successfully.")
```

### 4.5 GCP Secret Management (secretmanager.py)

**Function**: Strips high-risk fields such as passwords/Tokens.

```python
from aigear.secretmanager import SecretManager

secrets = SecretManager(project_id="ssc-ape-staging")

# Retrieve the latest private key version content
client_secret = secrets.get_secret_val(
    secret_id="MANJU_CLIENT_SECRET",
    secret_version="latest"
)
```

### 4.6 Third-Party Push Bridge (slack.py)

**Function**: (If business requires alerting) Sends operational notifications based on execution status.

```python
from aigear.slack import SlackNotifier

notifier = SlackNotifier(
    webhook_url="https://hooks.slack.com/services/YOUR/WEBHOOK",
    slack_on=True 
)

# Custom formatting for sending event reports
notifier.slack_webhook(
    message_content="Pipeline pipeline_version_1 fetch_data finished with 10k rows.",
    system_name="DataFetcher",
    pipeline_name="pipeline_version_1",
    step_name="fetch_data",
    event_type="success"  # success / warning / failure
)
```

### 4.7 Global Task Scheduler (task_scheduler.py)

**Function**: Engine entry point and serial distribution for specified Nodes. This is often the unified entry (Entrypoint) for the entire `Dockerfile` or cloud function.

```python
from aigear.task_scheduler import task_run

if __name__ == "__main__":
    # task_run() deeply parses command-line arguments sys.argv, routing the corresponding version and step to the respective src module for execution.
    task_run()
```

Execution method example (if via terminal):
```bash
python main.py --version pipeline_version_1 --step pipeline_version_1.fetch_store_list
```
